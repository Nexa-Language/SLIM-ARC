## 图 1：SLIM-ARC 三层整体架构（fig:architecture）

### 设计理念
非对称分层布局，打破"规整三横条"的常见模板。用错位的方块和斜向数据流箭头体现"协同"感。

### 布局描述
```
┌─────────────────────────────────────────────────────┐
│  量化优化层                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │
│  │ IQ4_XS  │  │ KV q4_0  │  │ FlashAttention   │  │
│  │ 40GB    │  │ 0.5x KV  │  │ IO-aware tiling  │  │
│  └────┬─────┘  └────┬─────┘  └────────┬─────────┘  │
└───────┼──────────────┼────────────────┼────────────┘
        ↓              ↓                ↓
┌───────┼──────────────┼────────────────┼────────────┐
│  运行时调度层         │                │            │
│  ┌────▼──────────────▼────────────────▼─────────┐  │
│  │  MoE Router Hook  ←  prefetch_scheduler     │  │
│  │       ↓                      ↑              │  │
│  │  unified_io_scheduler (bandwidth budget)    │  │
│  └───────────────────────┬─────────────────────┘  │
└──────────────────────────┼─────────────────────────┘
                           ↓ madvise(WILLNEED/RANDOM/DONTNEED)
┌──────────────────────────┼─────────────────────────┐
│  内核协同层              │                          │
│  ┌───────────────────────▼─────────────────────┐   │
│  │  mmap (45GB VSZ)  →  Linux Kernel           │   │
│  │  page cache  ←→  NVMe SSD (GGUF file)       │   │
│  │  demand paging / readahead control          │   │
│  └─────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

### 视觉风格
- 三层用不同浅色背景（蓝/绿/橙）
- 箭头用粗线+箭头帽，斜向连接体现数据流
- 右侧标注关键数据：45GB→2GB RSS、+71.4% decode、0.5× KV
- 底部 SSD 用圆柱体图标，CPU 用芯片图标

## 图 2：MADV_RANDOM 按需加载机制（fig:madv_random）

### 设计理念
左右对比，左侧"默认 WILLNEED"展示内存爆炸，右侧"MADV_RANDOM"展示精准加载。用"满 vs 空"的视觉对比强化 insight。

### 布局描述
```
左半（WILLNEED 默认）：              右半（MADV_RANDOM）：
┌─────────────────────┐             ┌─────────────────────┐
│   512 专家权重块     │             │   512 专家权重块     │
│  ┌─┐┌─┐┌─┐┌─┐┌─┐  │             │  ┌─┐  ┌─┐  ┌─┐     │
│  │■││■││■││■││■│  │ 全部加载    │  │ │  │ │  │ │     │ 仅 10 个
│  └─┘└─┘└─┘└─┘└─┘  │ ←──readahead│  └─┘  └─┘  └─┘     │ ←──page fault
│  ┌─┐┌─┐┌─┐┌─┐┌─┐  │             │  ┌─┐  ┌─┐           │   按需
│  │■││■││■││■││■│  │             │  │ │  │ │           │
│  └─┘└─┘└─┘└─┘└─┘  │             │  └─┘  └─┘           │
│       ... 45GB     │             │       ... 2GB       │
│   RSS = 45GB → OOM │             │   RSS = 2GB → OK    │
└─────────────────────┘             └─────────────────────┘
```

### 视觉风格
- 左侧所有方块填满深色（表示全部加载）
- 右侧只有 10 个方块填色（10/512），其余空心边框
- 底部用粗体大字标注 RSS 数值对比
- 中间用红色叉号 vs 绿色对号

## 图 3：MoE Router Hook 跨层专家预测（fig:moe_router_hook）

### 设计理念
时间轴从左到右，展示 Layer N → N+1 的预测-预取流水线。用"雷达扫描"隐喻。

### 布局描述
```
时间轴 →

Layer N 计算：                    Layer N+1 计算：
┌──────────────┐                 ┌──────────────┐
│ ffn_moe_topk │ ──缓存──→       │ 预取的专家    │
│ [e1,e2,..e10]│   expert IDs    │ 已在 page    │
└──────┬───────┘                 │ cache 中     │
       │                         └──────┬───────┘
       │ madvise(WILLNEED)              │ 直接计算
       ↓ on expert sub-regions          ↓
  ┌──────────────┐               ┌──────────────┐
  │ 内核异步预读  │               │ 无 page fault│
  │ 10 个专家     │               │ 延迟↓        │
  └──────────────┘               └──────────────┘
```

### 视觉风格
- 用虚线箭头表示"预测/预取"
- 专家块用 3D 合并张量的截面图（512 格，高亮 10 格）
- 中间放一个"雷达"图标表示预测

## 图 4：StreamingLLM KV Eviction（fig:kv_eviction）

### 设计理念
用"传送带"隐喻：KV token 在传送带上移动，sink 区固定不动，window 区滑动，中间被"丢弃"。

### 布局描述
```
KV Cache 时间线 →

┌───┬───┬───┬───┬──────────────────┬───┬───┬───┬───┐
│ S │ S │ S │ S │  ← EVICTED →     │ W │ W │ W │ W │
│ 1 │ 2 │ 3 │ 4 │  (old tokens)    │ n-2│n-1│ n │   │
└───┴───┴───┴───┴──────────────────┴───┴───┴───┴───┘
  ↑ sink (固定)    ↑ 中间被驱逐       ↑ sliding window
  4 tokens         ↑ 释放给权重缓存    最近 W tokens

内存：O(context_length) → O(sink + window) = O(1)
```

### 视觉风格
- sink 区用锁图标（固定）
- evicted 区用删除线+向下箭头（丢弃）
- window 区用滑动箭头
- 底部用内存占用条形图对比

## 图 5：FlashAttention IO-aware Tiling（fig:flashattn）

### 设计理念
对比"标准 attention"和"FlashAttention"的内存访问模式。用"分块"和"融合"的视觉。

### 布局描述
```
标准 Attention（3 次 I/O）：        FlashAttention（1 次 I/O，tiled）：
┌────┐  ┌────┐  ┌────┐            ┌──┬──┬──┬──┐
│ QK │→ │ S  │→ │ PV │            │QK│SP│PV│QK│ ... tiled
│    │  │    │  │    │            │  │  │  │  │   融合 kernel
└────┘  └────┘  └────┘            └──┴──┴──┴──┘
  ↑       ↑       ↑                 ↑ 一次 I/O
  3 次 DRAM 读写                    内存读写 ↓ 71%
```

### 视觉风格
- 左侧三个独立方块+箭头
- 右侧一个融合的大方块，内部分 tile 网格
- 用内存带宽的"水管"粗细对比

## 通用设计规范（NeurIPS/AAAI 标准）

1. **少文字多图**：每个图标/方块最多 2-3 个词，详细信息放图注
2. **不规整普通**：避免纯矩形堆叠，用圆角、斜向、错位打破规整
3. **颜色语义**：红色=问题/baseline，绿色=优化/solution，蓝色=机制，紫色=创新
4. **箭头语义**：实线=数据流，虚线=预测/控制，粗线=主要路径
5. **自包含图注**：图注应能独立解释图的含义，不依赖正文
6. **矢量图优先**：用 SVG/PDF 而非 PNG，保证缩放清晰
7. **字号**：图中文字 ≥ 10pt，图注 ≥ 9pt
