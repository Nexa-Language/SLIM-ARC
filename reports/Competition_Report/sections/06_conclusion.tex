\section{总结与未来展望}

\subsection{工作总结}

本文提出了 SLIM-ARC，一个面向端侧 MoE 大模型推理的操作系统级协同优化框架。通过系统性分析和实验验证，我们做出了以下核心贡献：

\begin{enumerate}
    \item \textbf{发现并利用了 MADV\_RANDOM 对 MoE decode 的加速效应}：通过四组单点消融实验，精确定位了 \texttt{posix\_madvise(MADV\_RANDOM)} 是 decode 阶段加速的核心驱动因素。其作用机制是：关闭内核顺序预读后，仅被 MoE Router 激活的专家权重页面（10/512 = 2\%）通过 page fault 按需加载，其余 98\% 的未激活专家权重完全不占用物理内存。

    \item \textbf{识别并消除了 GGML\_CPU\_REPACK 的内存翻倍问题}：发现 llama.cpp CPU 后端的 Q4\_K $\to$ q4\_K\_8x8 重打包机制会导致匿名内存翻倍（45\,GB $\to$ 90\,GB），是 80B 模型在受限环境下 OOM 的根因。通过 \texttt{-DGGML\_CPU\_REPACK=OFF} 禁用该机制，使模型从 OOM 转变为可运行。

    \item \textbf{构建了完整的优化技术栈}：将 IQ4\_XS 模型量化、KV Cache Q4\_0 量化、层感知预取调度器、MoE Router 跨层专家预测、FlashAttention、StreamingLLM KV Eviction 等优化技术有机集成，形成了从模型加载到运行时调度的完整优化链条，累计实现 64.5$\times$ 加速（0.08 $\to$ 5.16 t/s）。

    \item \textbf{实现了 80B 模型在端侧的流畅运行}：在 32\,GB 热缓存环境下，IQ4\_XS 量化的 Qwen3-Next-80B 达到了 \textbf{5.16\,tokens/s} 的 decode 速度（含 FlashAttention），实现了流畅的交互式推理。在 8\,GB 最受限环境下，达到了 0.76\,tokens/s，相比 baseline 提升 850\%（9.5 倍）。

    \item \textbf{通过标准 benchmark 验证了精度边界}：遵循综述~\cite{survey2026}的 ALEM 协议，在 GSM8K 上测量了不同量化配置的推理精度，发现 IQ4\_XS 虽保持语言流畅性但数学推理崩溃（0\%），而 Q4\_K\_M 保持 50\% 精度，为端侧模型选择提供了量化指导。
\end{enumerate}

\subsection{诚实评估与局限性}

\paragraph{prefetch\_scheduler 的冗余与 OS 底层绕开思路}
四组消融实验表明，在 80B 模型 8\,GB 环境下，prefetch\_scheduler 没有独立贡献。原因是：在有 MADV\_RANDOM 时，内核的 page fault 机制已足够按需加载，额外的 \texttt{madvise(WILLNEED)} 调用是冗余的；在无 MADV\_RANDOM 时，内核默认 WILLNEED 已全预读。深层原因是 MoE decode 的访存虽稀疏，但 NVMe 带宽（$\sim$3.5\,GB/s）远大于单层权重 I/O 需求，用户态预取的系统调用开销（每次 \texttt{madvise} 需陷入内核态）抵消了理论收益。

作为操作系统比赛项目，我们进一步思考了绕开用户态系统调用开销的底层方案：其一，使用 \texttt{userfaultfd}~\cite{cgroupsv2}在用户态注册缺页处理程序，将多个 page fault 批量处理，减少内核-用户态切换次数；其二，通过 \texttt{mmap(MAP\_POPULATE)} 或 \texttt{mlock} 在模型加载阶段预填充关键层的页面，避免 decode 期间的同步缺页；其三，直接调用 Linux 的 \texttt{readahead} 系统调用（而非 \texttt{madvise}），精确控制预读窗口大小和起始偏移，避免 \texttt{madvise} 的语义模糊性；其四，编写自定义内核模块拦截 \texttt{page\_fault} 路径，基于 MoE Router 输出在内核态直接触发 DMA 预读，完全绕开用户态调度。这些方案属于决赛阶段探索方向，初赛受时间限制未实现。

\paragraph{数据波动问题}
80B 模型在受限环境下的性能数据存在显著波动（同一配置 tg8 从 0.28 到 1.03）。这源于 45\,GB 模型与 8--32\,GB 物理内存的巨大差距，导致每次推理的 page cache 命中率高度依赖系统状态。虽然通过冷启动（drop\_caches）和多次测量缓解了部分波动，但这一问题在大模型受限推理中是固有的。

\paragraph{Speculative Decoding 的不适用}
测试 llama.cpp 内置 speculative decoding（ngram-simple）在 80B MoE 上呈现净负收益（-53.2\%），accept rate 仅 33.3\%。原因包括：无 small draft model（Qwen3-Next 无小版本）、MoE router 高熵增加不确定性、验证开销远超投机收益。这一结果与 unsloth 对 Qwen3.6-35B-A3B 的测试一致，证实了 speculative decoding 对 MoE 大模型的局限性。

\paragraph{KV Eviction 与 FlashAttention 的冲突}
实验发现 StreamingLLM eviction 与 FlashAttendance 同时启用时性能下降（pp64 12.99$\to$5.76 t/s），因为 eviction 的 \texttt{seq\_rm} 干扰了 FlashAttention 的 KV 布局。当前实现选择 FlashAttention 优先（因为收益更大），eviction 作为可选增强。

\subsection{未来展望}

由于初赛时间紧迫，SLIM-ARC 的部分工作尚未完善。在决赛阶段，我们将参考 \texttt{docs/others/赛题与学习路径.pdf} 和 \texttt{docs/others/INIT\_PROMPT.md} 中的初始规划，重点推进以下方向：完成 KV eviction manager 与推理流程的深度集成，实现基于注意力分数的 KV Block 级别驱逐和跨层投机预取（参考 SolidAttention~\cite{solidattention2026}的 block-wise sparsity 和 81\% 跨层选择相似性）；探索更激进的模型量化（Q2\_K/IQ2\_XXS，$\sim$22--28\,GB）以彻底消除 8\,GB 环境的数据波动；实现 Tile 级微流水线和融合反量化，将权重张量切分为对齐 L2/L3 cache 行的 Tile，I/O 线程与计算线程重叠执行；完成统一调度器在权重预取 + KV 换页 + 专家预取三路竞争下的协同验证；以及探索端侧 Agent 场景的语义缓存和 edge-cloud 协同（参考综述~\cite{survey2026}Sec 4.5 的 ECCC 框架），实现 84\% hit rate 的上下文摘要替代原始 KV。我们相信，通过操作系统虚拟内存机制与 MoE 稀疏性的深度协同，端侧大模型推理的“内存墙”问题将得到系统性解决。
