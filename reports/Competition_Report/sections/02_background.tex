\section{背景与相关工作}

\subsection{端侧 LLM 推理的内存墙挑战}

大语言模型（LLM）在自然语言理解、代码生成、多轮对话等任务上取得了突破性进展~\cite{brown2020gpt3, touvron2023llama, qwen3_2025}，但其庞大的参数规模与端侧设备有限的内存资源之间形成了严峻的矛盾。以本文使用的 Qwen3-Next-80B-A3B 模型为例~\cite{qwen3next2024}，其 Q4\_K\_M 量化后的权重大小为 45\,GB，而典型的端侧设备（如 Raspberry Pi 5 配备 8\,GB RAM，或中端智能手机配备 12--16\,GB RAM）远不足以容纳完整模型。即使采用更激进的 IQ4\_XS 量化（40\,GB），模型大小仍远超 8--16\,GB 的物理内存限制。这一矛盾被综述~\cite{survey2026}称为“内存墙”（Memory Wall）问题，是端侧 LLM 部署的首要瓶颈。

综述~\cite{survey2026}指出，LLM 推理过程涉及三类主要的内存消耗：

\begin{itemize}
    \item \textbf{模型权重}：占比最大（45\,GB for 80B Q4\_K\_M），在推理期间只读访问，是 I/O 带宽的主要消费者。
    \item \textbf{KV Cache}：随上下文长度线性增长，32K 上下文下可达 6\,GB（Qwen3-Next-80B，F16 KV），长上下文场景下可超过模型权重本身。
    \item \textbf{计算缓冲区}：包含 attention 中间结果、FFN 激活值等，通常 1--2\,GB，与 batch size 和序列长度相关。
\end{itemize}

当三类内存需求的总和超过物理内存时，操作系统会触发频繁的页面交换（swapping），导致推理速度急剧下降甚至完全不可用。综述 Sec 4.3 进一步指出，随着上下文长度从 4K 增长到 128K（已成为 2025 年模型的默认配置~\cite{survey2026}），KV Cache 的内存开销将主导总内存使用，使得即使 8B 参数的小模型也需要超过 16\,GB 内存仅用于 KV Cache。

\subsection{MoE 模型的稀疏性机遇}

混合专家模型（Mixture of Experts, MoE）为解决上述问题提供了独特机遇。MoE 架构~\cite{shazeer2017moe, fedus2022switch}通过将 FFN 层替换为多个并行的“专家”网络，并为每个 token 通过 Router 动态选择少量专家激活，实现了参数量与计算量的解耦。Qwen3-Next-80B-A3B~\cite{qwen3next2024}采用超稀疏 MoE 架构，包含 512 个专家网络，但每个 token 仅激活其中 10 个专家，稀疏率高达 98\%。这意味着：

\begin{equation}
    \text{有效激活权重} = \frac{10}{512} \times \text{总专家权重} \approx 2\% \times \text{总权重}
\end{equation}

理论上，如果能够仅加载被激活的专家权重，I/O 带宽需求可降低 98\%。然而，现有框架（如 llama.cpp~\cite{llamacpp2024}）将所有专家权重以合并的 3D 张量形式存储（如 \texttt{blk.\{l\}.ffn\_gate\_exps.weight}，形状为 \texttt{[in\_dim, hidden\_dim, n\_experts]}），无法在张量级别实现选择性加载。MobileMoE~\cite{mobilemoe2024}和 MoE-Prism~\cite{moeprism2024}的工作表明，MoE 的稀疏性如果在系统层面被充分利用，可以带来数量级的 I/O 带宽节省。综述 Sec 4.5 的 ECCC（Edge-Cloud Collaborative Computing）框架也强调了 MoE orchestration 在端侧场景的重要性。

\subsection{相关工作}

本节将现有相关工作分为四组：权重卸载与按需加载、KV Cache 管理与量化、MoE 稀疏性利用、Attention 优化与 KV Eviction。

\subsubsection{权重卸载与按需加载}

\paragraph{FlexInfer}
FlexInfer~\cite{flexinfer2025}是 EuroMLSys 2025 提出的端侧 LLM 推理框架，其核心机制包括：(1) 张量级异步预取，使用 Direct I/O 和多线程预取，在计算当前层时异步加载后续层的权重；(2) 均衡内存锁定，根据内存预算决定哪些层的权重常驻 RAM，哪些卸载到 SSD；(3) 灵活张量保留，静态启发式算法，根据总内存预算决定保留 FFN 还是 Attention 参数。FlexInfer 的局限在于：其 fork 版本的 GGUF 读取器与 Qwen3 架构不兼容；未利用 MoE 模型的专家稀疏性；未考虑 KV Cache 的内存优化。SLIM-ARC 在 FlexInfer 的“层感知预取”思路上发展，但用操作系统原生的 madvise 机制替代了 Direct I/O，简化了实现并利用了内核的 page cache。

\paragraph{llama.cpp 的 mmap 机制}
llama.cpp~\cite{llamacpp2024}原生支持 mmap 方式的模型加载，通过将 GGUF 文件映射到虚拟地址空间，利用内核的 demand paging 实现按需加载。然而，默认的 \texttt{madvise(WILLNEED)} 策略会触发内核的顺序预读（sequential readahead），对于 45\,GB 的大模型，预读行为会大量消耗物理内存页缓存，导致 8\,GB 环境下的 OOM。SLIM-ARC 的核心创新之一即识别了这一问题，并改用 \texttt{MADV\_RANDOM} 策略匹配 MoE 的稀疏访存模式。

\paragraph{PowerInfer / PowerInfer-2}
PowerInfer~\cite{powerinfer2024}在消费级 GPU 上实现了大模型推理，利用 ReLU 激活函数的稀疏性（约 90\% 神经元不激活）仅加载活跃神经元权重。PowerInfer-2~\cite{powerinfer2_2024}将这一思路推广到智能手机的 CPU+GPU 异构环境。虽然其场景是 GPU 混合，但其“利用稀疏性减少 I/O”的核心思路与 SLIM-ARC 一致。不同之处在于：PowerInfer 利用的是激活稀疏性（ReLU），SLIM-ARC 利用的是路由稀疏性（MoE Router）。

\subsubsection{KV Cache 管理与量化}

\paragraph{DUAL-BLADE}
DUAL-BLADE~\cite{dualblade2024}提出了 NVMe-direct 的 KV Cache 卸载方案，将注意力分数较低的 KV Block 异步驱逐到 SSD，并在需要时预取回 RAM。其核心思路是将 FlexInfer 的权重卸载框架泛化到 KV Cache。但 DUAL-BLADE 仅优化 KV Cache，未考虑权重与 KV 之间的 I/O 带宽竞争。SLIM-ARC 的统一 I/O 调度器部分借鉴了这一“带宽预算”思路。

\paragraph{HillInfer}
HillInfer~\cite{hillinfer2024}提出了基于 SmartSSD 的分层 KV Cache 驱逐策略，将 KV Cache 按注意力分数分为热、温、冷三级，冷数据卸载到 SmartSSD 的专用计算单元。其分层驱逐思路启发了 SLIM-ARC 的 sink + sliding window + cold tier 设计，但 SLIM-ARC 使用纯 CPU + NVMe 方案，无需专用硬件。

\paragraph{ScoutAttention}
ScoutAttention~\cite{scoutattention2024}提出了层前 CPU 预计算机制，利用 CPU 的空闲算力提前计算下一层的注意力分数，从而提前知道哪些 KV Block 需要被访问，实现精准的 KV Cache 预取。SLIM-ARC 的 MoE Router Hook 借鉴了这种“层前预测”思路，但将其应用于专家预测而非 KV 预测。

\paragraph{KVQuant / KIVI / ThinK}
KVQuant~\cite{kvquant2024}通过保留关键通道的高精度，实现了 2-bit KV 量化并支持 10M token 上下文。KIVI~\cite{kivi2024}提出非对称 KV 量化协议（Key 按通道、Value 按标记），无需调优即可部署。ThinK~\cite{think2024}基于注意力分数动态裁剪不重要的 KV 列。这些工作为 SLIM-ARC 的 KV q4\_0 量化提供了精度验证的理论基础——综述~\cite{survey2026}Sec 3.3 指出，4-bit KV 量化在多数任务上的精度损失可控。

\subsubsection{MoE 稀疏性利用}

\paragraph{MobileMoE}
MobileMoE~\cite{mobilemoe2024}提出了 MoE 专家缓存机制，通过轻量级 Router Predictor 提前 1--2 层预测即将激活的专家，并仅预取所需权重。理论分析表明，预测准确率达到 80\% 时可节省约 70\% 的带宽。但 MobileMoE 假设每个专家是独立张量，而 llama.cpp 中专家以合并 3D 张量存储，需要子区域级别的预取策略。SLIM-ARC 通过 \texttt{register\_expert\_tensor} 接口解决了这一问题。

\paragraph{MoE-Prism}
MoE-Prism~\cite{moeprism2024}提出了专家解耦策略，将大的合并专家张量拆分为独立的子张量，实现更灵活的调度。SLIM-ARC 采用了类似思路但更轻量：不修改 GGUF 格式，而是通过 \texttt{madvise} 对合并张量的子区域进行选择性预取。

\paragraph{eMoE / ProMoE}
eMoE~\cite{emoe2025}和 ProMoE~\cite{promoe2025}利用中间计算结果预测并预取未来专家权重。ProMoE 的 proactive caching 在多请求场景下显著降低 TTFT。这些技术补充了 SLIM-ARC 在单请求端侧场景的不足——端侧通常 batch=1，ProMoE 的多请求聚合优势无法发挥，但其“预测-预取”原理仍然适用。

\subsubsection{Attention 优化与 KV Eviction}

\paragraph{FlashAttention}
FlashAttention~\cite{flashattention2022, flashattention2_2023}通过 IO-aware tiling 融合，将 attention 的 QK$^T$ 和 softmax(P)V 合并为单个 kernel，避免中间矩阵的显存读写。综述 Sec 4.1 将其列为系统级优化的核心技术。llama.cpp 已内置 FlashAttention 实现（\texttt{-fa} 参数），SLIM-ARC 的实验验证了其在 80B decode 阶段带来 71.4\% 的加速。

\paragraph{StreamingLLM}
StreamingLLM~\cite{streamingllm2024}发现了“attention sink”现象：初始 token 累积了不成比例的高注意力权重，驱逐它们会导致 PPL 暴增。通过保留前 4 个 token（sink）+ 最近 W 个 token（sliding window），StreamingLLM 实现了恒定内存下的无限长流式推理。SLIM-ARC 将这一机制集成到 llama.cpp 的 decode 路径中。

\paragraph{SolidAttention}
SolidAttention~\cite{solidattention2026}（USENIX FAST'26）针对 AIPC（8-16\,GB DRAM）场景，通过 KV 交错布局、投机预取（跨层选择相似性 81\%）和 SSD-aware DAG 调度器，实现了 98\% 的 KV 内存减少和 3.1$\times$ 加速。其“block-wise 动态 attention sparsity”和“投机预取”思路为 SLIM-ARC 的后续优化提供了方向。

\paragraph{PagedAttention}
PagedAttention~\cite{pagedattention2023}（vLLM）将 KV Cache 组织为固定大小的 block，借鉴操作系统的虚拟内存分页机制，解决了 KV 碎片化问题。虽然 PagedAttention 主要面向服务器多请求场景，但其“分页管理 KV”的思路影响了端侧框架设计。综述 Sec 4.2 指出，PagedAttention 的架构思想正在影响 edge 框架。

\subsection{问题定义与期望解决方法}

基于上述分析，本文需要解决的核心问题为：\textbf{如何让 45\,GB 的 MoE 大模型在 8--16\,GB RAM 的端侧设备上高效运行？}

具体而言，需要同时解决以下子问题：

\begin{enumerate}
    \item \textbf{权重内存墙}：45\,GB 权重远超 8--16\,GB 物理内存，如何仅加载推理实际需要的权重？
    \item \textbf{MoE 稀疏性利用}：如何利用 98\% 的专家稀疏率，在系统层面实现选择性加载？
    \item \textbf{KV Cache 膨胀}：长上下文下 KV Cache 可达 6\,GB，如何控制其增长？
    \item \textbf{I/O 带宽竞争}：权重加载、KV 换页、专家预取三路 I/O 如何协调？
    \item \textbf{精度保持}：量化和驱逐策略如何避免损害模型推理能力？
\end{enumerate}

SLIM-ARC 的设计思路是：通过操作系统虚拟内存机制（mmap + madvise）利用 MoE 稀疏性解决子问题 1 和 2；通过 KV Cache 量化和 StreamingLLM eviction 解决子问题 3；通过统一 I/O 调度器解决子问题 4；通过 FlashAttention 和量化策略对比解决子问题 5。各技术的具体设计见第 3 节。
