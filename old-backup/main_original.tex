\section{实验原理}

\subsection{卷积神经网络中的卷积操作}

在卷积神经网络（CNN）中，卷积操作实质上是一种互相关运算（cross-correlation），而非信号处理中的严格卷积\cite{cs231n_conv_nets}。给定输入特征图$X$（尺寸$C_{in}\times H\times W$）和卷积核$K$（尺寸$C_{out}\times C_{in}\times k\times k$），输出特征图$Y$的第$(oc, oh, ow)$个元素计算公式为：

\begin{equation}
Y[oc, oh, ow] = \sum_{ic=0}^{C_{in}-1}\sum_{kh=0}^{k-1}\sum_{kw=0}^{k-1} X[ic, oh\cdot s - p + kh, ow\cdot s - p + kw] \cdot K[oc, ic, kh, kw]
\end{equation}

其中$s$为步幅（stride），$p$为填充（padding）。输出尺寸为：

\begin{equation}
out\_H = \lfloor(H + 2p - k) / s\rfloor + 1, \quad out\_W = \lfloor(W + 2p - k) / s\rfloor + 1
\end{equation}

本实验中固定$C_{in} = 3$，$C_{out} = 3$，$k = 3$，步幅分别取1、2、3，padding取$\lfloor k/2 \rfloor = 1$。

\subsection{直接卷积（滑窗法）}

直接卷积是最直观的实现方式。在GPU上，每个线程负责计算输出特征图中的一个元素。线程组织为2D线程块（$16\times 16$），通过blockIdx映射到不同的输出通道。

\begin{itemize}
    \item \textbf{基础版本}：直接从全局内存读取输入和卷积核数据
    \item \textbf{Shared Memory优化版本}：将当前输出通道对应的卷积核缓存到Shared Memory，减少对全局内存的重复访问
\end{itemize}

Shared Memory优化的收益在于：对于$C_{in}=3, k=3$的卷积核，每个线程块共享$3\times 3\times 3 = 27$个浮点数，避免了每个线程重复从全局内存读取这些值。

\subsection{im2col + GEMM方法}

im2col方法将卷积运算转化为矩阵乘法\cite{chellapilla2006high}，核心思路是将每个滑动窗口展开为一个列向量，所有窗口组成矩阵$X_{col}$（尺寸$C_{in}\times k\times k, out\_H\times out\_W$），同时将卷积核展开为行向量矩阵$W$（尺寸$C_{out}, C_{in}\times k\times k$），卷积运算变为：

\begin{equation}
Y = W \times X_{col}
\end{equation}

这种方法的优势在于：
\begin{itemize}
    \item 将不规则的卷积访问转化为规则的矩阵乘法，可复用高度优化的GEMM实现（如cuBLAS）
    \item GEMM的计算密度高，能充分利用GPU的并行计算单元
\end{itemize}

代价是需要额外的内存存储im2col展开后的矩阵，内存占用为原始输入的$k^2$倍。

\subsection{cuDNN卷积}

cuDNN是NVIDIA官方的深度神经网络加速库\cite{nvidia_cudnn_developer_guide}，其卷积实现融合了多种优化技术：
\begin{itemize}
    \item 自动选择最优卷积算法（通过\texttt{cudnnGetConvolutionForwardAlgorithm\_v7}查询）\cite{nvidia_cudnn_api_ref}
    \item 支持Tensor Core加速（FP16计算 + FP32累加）
    \item 支持多种算法：隐式GEMM、Winograd变换\cite{lavin2016fast}、FFT等
    \item 自动处理数据布局转换（NCHW $\leftrightarrow$ NHWC）
\end{itemize}

\section{实验实现}

\subsection{任务1：CUDA直接卷积}

直接卷积Kernel的核心设计如下：每个线程计算输出的一个元素，通过线程块和网格的映射覆盖整个输出张量。

\begin{lstlisting}[caption={直接卷积Kernel（Shared Memory优化版）}]
__global__ void direct_conv_kernel_smem(
    const float* __restrict__ input,
    const float* __restrict__ kernel,
    float* __restrict__ output,
    int H, int W, int C_in, int C_out,
    int K, int stride, int padding,
    int out_H, int out_W)
{
    const int TILE_H = 16, TILE_W = 16;
    int ow = blockIdx.x * TILE_W + threadIdx.x;
    int oh = blockIdx.y * TILE_H + threadIdx.y;
    int oc = blockIdx.z;

    extern __shared__ float smem[];
    float* s_kernel = smem;
    int kernel_size = C_in * K * K;
    int tid = threadIdx.y * TILE_W + threadIdx.x;
    int block_threads = TILE_H * TILE_W;

    // 协作加载卷积核到Shared Memory
    for (int i = tid; i < kernel_size; i += block_threads)
        s_kernel[i] = kernel[oc * kernel_size + i];
    __syncthreads();

    if (ow >= out_W || oh >= out_H) return;

    float sum = 0.0f;
    for (int ic = 0; ic < C_in; ic++)
        for (int kh = 0; kh < K; kh++)
            for (int kw = 0; kw < K; kw++) {
                int ih = oh * stride - padding + kh;
                int iw = ow * stride - padding + kw;
                if (ih >= 0 && ih < H && iw >= 0 && iw < W)
                    sum += input[ic*H*W + ih*W + iw]
                         * s_kernel[ic*K*K + kh*K + kw];
            }

    output[oc * out_H * out_W + oh * out_W + ow] = sum;
}
\end{lstlisting}

线程块配置为$16\times 16 = 256$线程，网格维度为$\lceil out\_W/16\rceil \times \lceil out\_H/16\rceil \times C_{out}$。

\subsection{任务2：im2col + GEMM卷积}

实现分为两步：首先用im2col Kernel将输入展开，然后调用cuBLAS的\texttt{cublasSgemm}执行矩阵乘法。

\begin{lstlisting}[caption={im2col Kernel}]
__global__ void im2col_kernel(
    const float* __restrict__ input,
    float* __restrict__ col,
    int H, int W, int C_in,
    int K, int stride, int padding,
    int out_H, int out_W)
{
    int col_height = C_in * K * K;
    int col_width = out_H * out_W;
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx >= col_height * col_width) return;

    int col_row = idx / col_width;  // (ic, kh, kw)
    int col_col = idx % col_width;  // (oh, ow)

    int kw = col_row % K;
    int kh = (col_row / K) % K;
    int ic = col_row / (K * K);
    int ow = col_col % out_W;
    int oh = col_col / out_W;

    int ih = oh * stride - padding + kh;
    int iw = ow * stride - padding + kw;

    col[col_row * col_width + col_col] =
        (ih >= 0 && ih < H && iw >= 0 && iw < W)
        ? input[ic*H*W + ih*W + iw] : 0.0f;
}
\end{lstlisting}

矩阵乘法使用cuBLAS的行主序调用方式：$Y = W \times X_{col}$，其中$W$为$C_{out}\times(C_{in}\cdot K^2)$，$X_{col}$为$(C_{in}\cdot K^2)\times(out\_H\cdot out\_W)$。

\subsection{任务3：cuDNN卷积}

使用cuDNN的\texttt{cudnnConvolutionForward} API，关键步骤如下：

\begin{lstlisting}[caption={cuDNN卷积调用关键步骤}]
// 设置卷积描述
cudnnSetConvolution2dDescriptor(conv_desc,
    padding, padding, stride, stride, 1, 1,
    CUDNN_CROSS_CORRELATION, CUDNN_DATA_FLOAT);

// 查询最优算法

\end{lstlisting}

\section{性能分析}

\subsection{测量方法与数据波动}

在性能测量过程中，我们采用CUDA事件计时（\texttt{cudaEvent}）记录Kernel执行时间，每次测量进行10次迭代取平均值，前3次作为预热不计入统计。尽管如此，GPU性能测量仍存在一定的数据波动，主要原因包括：

\begin{itemize}
    \item \textbf{GPU动态频率调节}：现代GPU会根据负载和温度自动调整时钟频率，导致相同计算的时间存在波动
    \item \textbf{缓存状态差异}：首次执行与后续执行的缓存命中率不同，影响内存访问延迟
    \item \textbf{系统背景负载}：操作系统和其他进程可能干扰GPU调度和内存带宽
    \item \textbf{小尺寸数据的测量误差}：当Kernel执行时间接近计时器分辨率（约0.5μs）时，相对误差会显著增大
\end{itemize}

特别是在小尺寸输入（如32×32、64×64）和高stride值的情况下，Kernel执行时间极短（<0.1ms），数据波动更为明显。例如在某些配置下，Shared Memory优化版本的表现可能不如基础版本，这很可能是测量误差或GPU状态波动所致，而非算法本身的性能差异。在分析结果时，我们更关注大规模输入（≥256）下的趋势，此时测量结果更加稳定可靠。

\subsection{直接卷积 vs im2col+GEMM}

在stride=1的情况下，直接卷积在大输入尺寸（$\geq 256$）时明显优于im2col+GEMM。以4096$\times$4096为例：
\begin{itemize}
    \item RTX 4060：直接卷积 6.70ms vs im2col+GEMM 25.79ms，快3.85倍
    \item A100：直接卷积 3.26ms vs im2col+GEMM 7.03ms，快2.16倍
\end{itemize}

这个结果有些出乎意料，因为通常认为im2col+GEMM应该更快。分析原因如下：

\begin{enumerate}
    \item \textbf{im2col的内存开销}：im2col需要将输入展开为$k^2 = 9$倍的矩阵，导致额外的内存分配和写入开销。对于$3\times 3$的小卷积核，这个开销相对较大。
    \item \textbf{GEMM矩阵维度不友好}：卷积转化为GEMM后，矩阵$W$仅为$3\times 27$，维度太小，无法充分利用cuBLAS的高度优化路径。cuBLAS在大矩阵乘法时才能发挥最大性能。
    \item \textbf{Kernel启动开销}：im2col+GEMM需要两次Kernel调用（im2col变换+GEMM），而直接卷积只需一次。
\end{enumerate}

\subsection{stride对性能的影响}

步幅增大时，输出尺寸减小，导致：
\begin{itemize}
    \item \textbf{计算量减少}：stride=2时FLOPS降为stride=1的约1/4，stride=3时约1/9
    \item \textbf{GPU利用率下降}：输出变小意味着活跃线程减少，GPU利用率降低
    \item \textbf{Kernel启动开销占比增大}：计算量减少后，固定的Kernel启动开销占比变大
\end{itemize}

stride=3时，所有方法的性能差距都在缩小，因为输出尺寸已经很小，计算被各种固定开销主导。

\subsection{自实现 vs cuDNN}

在两个平台上，直接卷积在小到中等规模输入时均快于cuDNN。这个现象值得注意：

\begin{itemize}
    \item cuDNN的卷积实现为通用性做了很多妥协：支持任意数据格式、分组卷积、dilation等。这些通用性带来了额外的判断和分支开销。
    \item 对于$C_{in}=3, k=3$这种小通道小核的配置，cuDNN的高度优化路径（如Winograd、FFT）可能反而不如简单的滑窗法高效。
    \item cuDNN在大规模输入（$\geq 2048$）和stride较大时开始展现优势，说明其优化在计算密集的场景下更有效。
\end{itemize}

\subsection{RTX 4060 vs A100 性能差异}

A100在计算密集型任务上的优势明显：
\begin{itemize}
    \item 4096$\times$4096 stride=1：A100直接卷积 3.26ms vs RTX 4060 6.70ms，快2.06倍
    \item A100拥有108个SM vs RTX 4060的24个SM，计算吞吐量约为4.5倍
    \item 但实际加速比不到理论值，因为小核卷积（3$\times$3）的计算密度不够高，内存访问成为瓶颈
\end{itemize}

\subsection{改进方向}

如果要进一步提升自实现卷积的性能，可以考虑：
\begin{enumerate}
    \item \textbf{使用Tensor Core}：将卷积转化为WMMA (Warp Matrix Multiply Accumulate)操作，利用FP16计算和FP32累加
    \item \textbf{Winograd变换}：对于$3\times 3$卷积，Winograd F(2$\times$2, 3$\times$3)可以将乘法次数减少2.25倍
    \item \textbf{数据预取}：使用异步拷贝（\texttt{cp.async}）将全局内存数据预取到Shared Memory
    \item \textbf{融合操作}：将卷积与后续的BatchNorm、ReLU等操作融合为一个Kernel，减少内存读写
\end{enumerate}

\subsection{关于Nsight工具的使用说明}

本实验原计划使用Nsight Systems进行详细的Kernel级性能分析，但由于实验环境的限制（WSL2中的GPU profiling支持有限），未能成功运行\texttt{nsys}和\texttt{ncu}工具。作为替代，我们通过以下方式进行了性能分析：

\begin{itemize}
    \item 使用\texttt{nvidia-smi}监控GPU利用率和显存带宽
    \item 通过多平台对比（RTX 4060 vs A100）分析硬件特性对性能的影响
    \item 基于GFLOPS计算效率，推断计算密集度和内存访问模式
\end{itemize}

如果有机会在原生Linux环境下运行Nsight工具，我们期望能够观察到：
\begin{itemize}
    \item 直接卷积Kernel的内存访问模式（全局内存vs共享内存的带宽利用率）
    \item 不同线程块配置对SM占用率的影响
    \item im2col变换的内存写入带宽瓶颈
\end{itemize}

\section{实验心得与总结}

通过本次实验，我对GPU并行计算和卷积优化有了更深入的理解。

\textbf{实践中的发现：}
\begin{itemize}
    \item \textbf{小即是美}：对于$3\times 3$这种小卷积核，简单的滑窗法反而比复杂的im2col+GEMM更快。这让我意识到，算法选择必须考虑具体的问题规模，"通用最优"并不存在。
    \item \textbf{库的代价}：cuDNN作为工业级库，在小规模问题上表现不如手写代码，这说明库的通用性是有代价的。但在大规模场景下，cuDNN的优势逐渐显现，这可能是因为其内部针对大规模问题做了专门优化。
    \item \textbf{测量是一门艺术}：GPU性能测量的波动比我预想的大得多。特别是在小尺寸场景下，数据的不稳定性让我意识到，基准测试需要更严谨的方法论。
    \item \textbf{硬件差异显著}：A100相比RTX 4060有2-3倍的加速，但远未达到理论上的4.5倍差距。这说明小卷积核的计算密度不够，内存带宽成为瓶颈。
\end{itemize}

\textbf{主要技术结论：}
\begin{enumerate}
    \item 对于小通道、小卷积核的卷积，简单的滑窗法配合Shared Memory优化就能达到很好的性能，甚至超过cuDNN。
    \item im2col+GEMM方法在矩阵维度太小时反而受限于GEMM的启动开销，不适合$3\times 3$这种小核场景。
    \item stride增大显著降低GPU利用率，因为输出尺寸减小导致计算量不足。
    \item A100相比RTX 4060在计算密集场景下约有2$\sim$3倍的加速，但在kernel启动开销占主导的小规模场景下差距缩小。
    \item 进一步优化的方向包括Winograd变换、Tensor Core加速和操作融合。
\end{enumerate}

\textbf{未来工作方向：}
如果时间允许，我计划进一步探索：
\begin{itemize}
    \item 使用Winograd变换优化$3\times 3$卷积，预期可减少2.25倍的乘法操作
    \item 实现Tensor Core加速，利用FP16计算和FP32累加提升吞吐量
    \item 探索更大通道数（16、32、64）和更大卷积核（5×5、7×7）的性能特征
    \item 使用Nsight工具进行详细的内存访问模式分析
\end{itemize}
